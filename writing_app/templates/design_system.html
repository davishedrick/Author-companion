Buttons

Choose button weight by decision importance, not visual variety.

Primary plus secondary

Choose this when one action commits the current flow and the other is a safe alternative. It creates a clear path forward without hiding the escape hatch.

Compact row actions

Choose this inside history rows, issue rows, and project lists where the surrounding item already supplies context. It keeps repeated actions scannable and low-noise.

Route and mode chips

Choose this for switching sections or modes rather than submitting data. It makes navigation feel tactile while staying smaller than a full command button.

Cards

Cards organize work units; choose the quietest frame that still helps scanning.

Single stat card

Choose this when one manuscript fact should lead a compact summary. Keep the label short, make the value clear, and use one quiet verification line.

Priority pills

Choose pills only for priority components where severity changes triage order. The color and compact shape help high, medium, and low scan quickly in repeated issue lists.

Forms

Forms should pair visible labels, compact grouping, and clear final actions.

Paired short fields

Choose this for project metadata that users can review side by side. It reduces vertical churn while preserving explicit labels.

Full-width narrative field

Choose this when the answer is prose, notes, or context. It gives the writer room to think without cramming text into a metric-style field.

Goal inputs with final action

Choose this when structured values culminate in one save decision. The right-aligned action row makes review-before-commit feel deliberate.

Progress

Use progress rails only for one measurable target with a known percent. Counts, checkpoints, and categorical states need separate list, stepper, or heatmap patterns.

Simple completion rail

Choose this for a single measurable target, such as manuscript completion or a monthly word goal. The label names the target, the percentage states the value, and the rail visualizes the same value once.

Navigation

Navigation should make location, mode, and project context legible.

Sidebar route list

Choose this only where the existing workspace shell still uses the sidebar pattern. It gives persistent location cues for dense app areas.

Header route chips

Choose this for the current compact global navigation. It keeps primary destinations accessible without consuming the full left rail.

Project context selector

Choose this when navigation depends on the active project. It prevents workspace actions from feeling detached from the manuscript they affect.

Modals

Dialogs block the flow only when the next choice changes data, recovery, or tracking state.

Compact confirmation dialog

Choose this over a wide dialog when the user is confirming one known consequence. It keeps the interruption brief and keeps the decision focused.
